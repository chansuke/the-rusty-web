About

This project demonstrates a complete, albeit simple, example of integrating Rust code into a web application. This is accomplished by compiling Rust to asm.js or WebAssembly. The basic design pattern this site explores uses Rust to implement CPU bound portions of an app while using existing web technologies to handle user facing, I/O bound pieces. The guide explores this design pattern in detail.

The demo compares implementations of the k-means clustering algorithm (this example's CPU bound task) applied to an image. The algorithm gives the resulting image a softened, painted look. Setting the k parameter controls the number of colors and the steps parameter controls the algorithm's number of iterations. After running the demo a few times, see the results section to compare runtimes.

Whereas this project looks at runtime performance, there are other benefits this design pattern offers: reducing website asset size, reducing JavaScript "compile" times, and the ability to use Rust!

Note: The demo may lock up the page. This is especially true on mobile devices. Consider uploading a smaller image and keeping the k and steps parameters small. In a real application, the task would either need to be broken into smaller pieces or run in a background thread using Web Workers.

*** Disclaimer ***

These are all relatively young technologies. Many of the benefits offered by this design pattern are not yet fully realized. While this project presents performance information, the actual performance numbers may not live up to expectations. They did not live up to mine. That being said, the primary purpose of this project is not to benchmark performance, but is to provide a guide to using these technologies as they develop.

Benchmarks

The benchmarks show the average step duration computed for ten steps with k values ranging from two to sixty-four (smaller values are better).

k=2 (ms) k=4 (ms) k=8 (ms) k=16 (ms) k=32 (ms) k=64 (ms)
Firefox 54 Rust (wasm) 314 528 878 1653 3179 6180
Chromium 58 Rust (asmjs) 149 228 372 665 1257 2484
Chromium 58 Rust (wasm) 134 212 365 651 1219 2326
Chromium 58 JavaScript 195 220 299 474 895 1606
Firefox 54 JavaScript 101 140 229 384 679 1257
Firefox 54 Rust (asmjs) 86 103 127 179 292 502
Native Rust 22 28 38 61 99 177

Overall, I would say the benchmarks are confussing and disappointing. While the asm.js version in Firefox is very promising, it is overshadowed by the Firefox WebAssembly version performing considerably worse than all other benchmarks. This is odd because the WebAssembly version beats the asm.js version in Chromium, but unfortunately both lose out to the plain old JavaScript implementation. I tried to keep the implementations of the algorithms as similar, yet idiomatic, as possible to present a fair comparison.

I can think of a few explanations for the odd results. First, there is only one hot section in the code: calculating the distance between two RGB colors using euclidean distance. Modern JavaScript engines are probably pretty good at optimizing for this simple computation. In fact, I rewrote this function to use an imperitive instead of functional style and saw a drastic speedup in the JavaScript version. Secondly, the asm.js code produced by the Rust compiler uses the "almost asm" directive opposed to the "use asm" directive in the resulting JavaScript. I believe this may have an impact on how much browsers optimize for asm.js. Finally, WebAssembly just reached its minimal viable product release making it a fair bet that browsers are far from having optimal implementations.

Despite the results, I think the big take away is that modern JavaScript is fast, but there is still a definate gap between it and native speeds. We have technology that is proof that this gap can be closed (the asm.js version in Firefox). Only the future and hard work can tell us how much more this gap can be closed.
